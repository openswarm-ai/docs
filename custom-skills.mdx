## Overview

Skills in OpenSwarm are **not code or API integrations** — they're structured **Markdown instruction files** that teach an agent how to approach a specific type of task. Think of a skill as expert-level instructions you'd give to a capable assistant: *"When someone asks you to do X, here's exactly how to think about it and what to do."*

Custom skills let you capture any workflow, domain knowledge, or specialized process and make it available to your agents on demand.

---

## What a Skill Actually Is

A skill is a directory containing a `SKILL.md` file — a Markdown document with YAML frontmatter. That's it. No servers, no deployments, no code compilation.

```
my-skill/
├── SKILL.md              ← Required: the core instructions
└── (optional resources)
    ├── scripts/           ← Executable code for repetitive tasks
    ├── references/        ← Docs loaded into context as needed
    └── assets/            ← Templates, icons, fonts, etc.
```

The `SKILL.md` has two parts:

### 1. Frontmatter (Metadata)

```yaml
---
name: pdf-processor
description: "Use this skill whenever the user wants to do anything with PDF files.
  This includes reading, merging, splitting, rotating, watermarking, encrypting,
  or creating PDFs. If the user mentions a .pdf file or asks to produce one,
  use this skill."
---
```

- **`name`** — The skill's identifier
- **`description`** — The most important field. This is what the agent reads to decide whether to activate the skill. It should clearly describe both *what* the skill does and *when* to use it.

### 2. Body (Instructions)

The rest of the file is Markdown — the actual instructions the agent follows when the skill is triggered. This can include:
- Step-by-step workflows
- Output format templates
- Examples with input/output pairs
- References to bundled scripts or docs
- Edge cases and decision trees

## How Skills Get Triggered

Skills in OpenSwarm use a **three-level loading system**:

1. **Metadata** (name + description) — Always in the agent's context (~100 words). The agent reads this to decide if the skill is relevant.
2. **SKILL.md body** — Loaded into context when the skill triggers. Keep this under ~500 lines.
3. **Bundled resources** — Loaded on demand. Scripts can execute without being read into context.

When a user sends a message, the agent scans the descriptions of all available skills. If a description matches the user's intent, the agent activates that skill and follows its instructions.

<Note>
The description is the primary triggering mechanism. A skill with a vague description won't get used — even if its body instructions are perfect. Write descriptions that are specific and slightly "pushy" about when to trigger, because agents tend to under-trigger rather than over-trigger.
</Note>

## Building a Custom Skill

### Step 1: Define Intent

Before writing anything, answer these questions:
- What should this skill enable the agent to do?
- What user messages or contexts should trigger this skill?
- What's the expected output format?
- Does the skill need external scripts or reference docs?

### Step 2: Write the SKILL.md

Here's a real example — a skill that creates Word documents:

```markdown
---
name: docx
description: "Use this skill whenever the user wants to create, read, edit, or
  manipulate Word documents (.docx files). Triggers include: any mention of
  'Word doc', 'word document', '.docx', or requests to produce professional
  documents with formatting like tables of contents, headings, page numbers,
  or letterheads. Do NOT use for PDFs, spreadsheets, or Google Docs."
---

# Word Document Creator

This skill handles all .docx file operations.

## Creating a new document

When the user wants to create a Word document:

1. Determine the document type (report, letter, memo, etc.)
2. Use the `python-docx` library to generate the file
3. Apply professional formatting:
   - Use heading styles for structure
   - Add page numbers in the footer
   - Include a table of contents if the doc has 3+ sections

## Output format

Always save files with descriptive names (not "output.docx").
Confirm the file path to the user when done.

## Example

**Input:** "Create a project status report for Q4 with sections for
progress, blockers, and next steps"

**Output:** A formatted .docx file with heading hierarchy, proper
styles, and placeholder content the user can fill in.
```

### Step 3: Add to OpenSwarm

Skills are stored at `~/.claude/skills/`. To add your skill:

1. Go to the **Skills** page in OpenSwarm
2. Click the **+** button to create a new skill
3. Fill in:
   - **Name** — your skill's name
   - **Description** — the trigger description (this maps to the frontmatter `description`)
   - **Command** — a slash command identifier (e.g., `docx`)
   - **Content** — the full Markdown body of your skill
4. Click **Create Skill**

Your skill is now saved as a `.md` file in `~/.claude/skills/` and immediately available to agents.

## Bundling Resources

For complex skills, the `SKILL.md` body alone may not be enough. You can bundle additional files:

### Scripts
Executable code for tasks that should be deterministic — things the agent would otherwise reinvent every time.

```
my-skill/
├── SKILL.md
└── scripts/
    └── create_chart.py    ← Agent can execute this directly
```

In your `SKILL.md`, reference it:
```markdown
## Chart generation
Use the bundled script at `scripts/create_chart.py` to generate charts.
Do not write your own charting code.
```

**When to bundle a script:** If you notice the agent writing the same helper code across multiple test cases, that's a strong signal to bundle it.

### References
Documentation files the agent reads on demand — useful when you need domain-specific details but don't want to bloat the main `SKILL.md`.

```
cloud-deploy/
├── SKILL.md              ← Workflow + selection logic
└── references/
    ├── aws.md            ← Agent reads only the relevant one
    ├── gcp.md
    └── azure.md
```

For large reference files (>300 lines), include a table of contents so the agent can navigate efficiently.

### Assets
Static files used in output — templates, images, fonts. The agent uses these but doesn't need to read them into context.

## Skill Registry

OpenSwarm includes a built-in **Skill Registry** — a browsable catalog of community skills sourced from the [Anthropic Skills repository](https://github.com/anthropics/skills). You can:

- **Browse** skills organized by category in the Skills page sidebar
- **Search** across all registry skills by name or description
- **Install** any registry skill with one click — it copies to your local `~/.claude/skills/`
- **Edit & Install** — fork a registry skill, customize it, then install your version

Registry skills work identically to custom skills you write — the only difference is where they came from.

## Writing Effective Skills

### Description Matters Most

The description is the #1 factor in whether your skill gets triggered. Bad description = unused skill, no matter how good the instructions are.

| ❌ Bad | ✅ Good |
|--------|---------|
| "Helps with data" | "Use this skill whenever the user wants to analyze CSV files, generate pivot tables, compute statistics, or visualize data from tabular datasets. Trigger for any mention of .csv, spreadsheets, or data analysis." |

### Instructions: Explain the Why

Agents are smart — they respond better to understanding *why* something matters than to rigid `ALWAYS` / `NEVER` rules. Instead of:

```
ALWAYS use heading level 2 for sections. NEVER use heading level 1.
```

Write:
```
Use heading level 2 for sections — level 1 is reserved for the document
title, so using it for sections breaks the document hierarchy and makes
screen readers misinterpret the structure.
```

### Keep It Focused

One skill should handle one domain. Don't create an "everything skill" — keep skills scoped so they trigger precisely. If a skill is growing past ~500 lines, split it into a concise `SKILL.md` with pointers to reference files.

### Use Examples

Include concrete input/output examples. They're one of the most effective ways to align the agent's behavior with what you actually want.

```markdown
## Commit message format

**Example 1:**
Input: Added user authentication with JWT tokens
Output: feat(auth): implement JWT-based authentication

**Example 2:**
Input: Fixed crash when email field is empty
Output: fix(auth): handle empty email in login form
```

## Sharing Skills

Custom skills you build can be:
- **Reused across agents** — any agent in your workspace can access installed skills
- **Shared with your team** — skills in your workspace are available to all team members
- **Contributed to the community** — publish to the [Anthropic Skills repo](https://github.com/anthropics/skills) for others to install from the registry
